# Feature 04.6: Dark Mode

## Goal

Let users choose their preferred colour scheme ŌĆö light, dark, or follow the OS ŌĆö and persist that choice across sessions.

## Status

Implemented.

## Scope

In scope:

- System-preference detection via `prefers-color-scheme` media query.
- Three-way theme toggle: Light / System / Dark.
- Preference persisted in `localStorage` (`snapvote-theme`).
- `ThemeProvider` context applies `.dark` class to `<html>` so all Tailwind `dark:` variants activate.
- Dark variants added to all shared UI primitives (`Card`, `Button`, `Input`, `Label`).
- Dark variants added to all pages and layout shells (`AuthShell`, `SessionGate`, auth pages, poll pages, home page).
- Dark version of the `.auth-canvas` gradient background.
- `ThemeToggle` component (Sun / Monitor / Moon icons) ŌĆö placed in the Navbar for authenticated pages and fixed top-right for auth / join pages.

Out of scope:

- Per-route or per-component theme overrides.
- High-contrast / accessibility themes.
- Server-side persistence of the theme preference (user account setting).
- Animated theme transition effects.

## User Journeys

1. First-time visitor lands on the login page ŌĆö the app respects the OS dark/light setting immediately with no flash.
2. User clicks the theme toggle and selects Dark ŌĆö the entire app switches to dark mode and the choice survives a page refresh.
3. User selects System ŌĆö the app tracks OS changes in real time (e.g. macOS scheduled dark mode at sunset).
4. User selects Light ŌĆö the app stays light regardless of OS setting.

## Acceptance Criteria

- On first load with no saved preference, the app matches `prefers-color-scheme`.
- Changing the OS preference while the app is open updates the resolved theme immediately when preference is `'system'`.
- Selecting a manual preference overrides the system preference and is reflected instantly.
- The selected preference survives a hard refresh (persisted in `localStorage`).
- `ThemeToggle` clearly indicates the currently active option.
- All pages and reusable components are legible and visually consistent in both light and dark modes.
- No flash of unstyled / wrong-theme content on initial load.

## API Touchpoints

None ŌĆö this is a pure frontend feature.

## Data Touchpoints

- `localStorage` key: `snapvote-theme` ŌĆö stores `'light' | 'dark' | 'system'`.

## Frontend Touchpoints

- New `apps/client/src/context/theme.context.tsx` ŌĆö `ThemeProvider`, `useTheme` hook, resolved-theme logic.
- Updated `apps/client/src/main.tsx` ŌĆö wraps `<App>` with `<ThemeProvider>`.
- New `apps/client/src/components/ui/theme-toggle.tsx` ŌĆö three-way icon button group.
- Updated `apps/client/src/styles.css`:
  - `@custom-variant dark (&:where(.dark, .dark *))` for class-based dark mode (Tailwind v4).
  - Dark variant of `.auth-canvas` gradient.
  - `color-scheme` declaration per theme.
- Updated UI primitives with `dark:` variants: `card.tsx`, `button.tsx`, `input.tsx`.
- Updated layout shells with `dark:` variants: `auth-shell.tsx`, `session-gate.tsx`, `navbar.tsx`, `app-shell.tsx`.
- Updated all pages with `dark:` variants: `home.page.tsx`, `login.page.tsx`, `register.page.tsx`, `forgot-password.page.tsx`, `reset-password.page.tsx`, `create-poll.page.tsx`, `poll-detail.page.tsx`, `poll-join.page.tsx`.

## Testing

- 11 unit tests for `ThemeProvider` covering:
  - Default to `'system'` when no localStorage value is set.
  - Resolves to `'dark'` when system pref is dark and theme is `'system'`.
  - Resolves to `'light'` when theme is manually set to `'light'` regardless of system pref.
  - Adds `.dark` class to `document.documentElement` when resolved theme is dark.
  - Removes `.dark` class when resolved theme switches to light.
  - Persists choice to localStorage.
  - Reads persisted choice from localStorage on mount.
  - Tracks system preference changes in real time when on `'system'`.
  - Does NOT track system changes when a manual theme is set.
  - Throws if `useTheme` is used outside `ThemeProvider`.

## Technical Debt

Current debt:

- All existing Tailwind classes were light-mode only; dark: variants have now been applied.

Planned debt risk:

- New pages added in future features will need `dark:` variants added manually.

Mitigation:

- Follow the colour-token convention below when adding new UI.

## Colour Conventions

| Role             | Light                                               | Dark                                                                      |
| ---------------- | --------------------------------------------------- | ------------------------------------------------------------------------- |
| Page / canvas bg | `auth-canvas` (CSS gradient)                        | `dark .auth-canvas` (dark gradient)                                       |
